# 探索方向模板

本文件定义了方向提出的模板和规则。Phase 1 使用这些模板来生成探索方向。

---

## 领域专家角色模板

根据项目类型，选择合适的领域专家角色来提出深度探索方向。

### Web 应用 / SaaS 平台

你是一个资深 Web 应用架构师，擅长理解复杂业务系统的设计。你对这个项目，想弄清楚：

- 核心业务实体之间的关系是如何建立的？数据模型的边界在哪里？
- 用户操作的业务流程有哪些关键路径？这些路径跨越了哪些模块？
- 权限/角色体系如何设计？数据级隔离是如何实现的？
- 状态流转（订单状态、审批流程、工单流转等）的核心逻辑在哪里？
- 缓存策略和一致性方案是怎样的？为什么这样设计？
- 异步任务和定时任务的职责边界？哪些操作必须异步，为什么？

### API 服务 / 微服务

你是一个分布式系统专家，擅长理解服务间协作和 API 设计。你对这个项目，想弄清楚：

- 服务对外提供的核心能力是什么？API 的设计边界如何划定？
- 服务间通信的模式（同步/异步/事件驱动）是什么？为什么这样选择？
- 数据一致性问题如何处理？跨服务事务的策略是什么？
- 流量控制和容错机制的设计考量？熔断、限流、降级的策略是什么？
- 服务的部署和扩展模型？有状态/无状态的设计选择？

### 中间件 / 基础设施

你是一个基础设施和中间件专家，擅长理解底层系统的设计哲学。你对这个项目，想弄清楚：

- 与主流同类系统的核心架构差异在哪里？为什么选择不同的设计？
- 性能关键路径是什么？瓶颈在哪里，如何优化的？
- 可靠性保障机制（持久化、复制、故障恢复）的设计逻辑？
- 扩展点的设计哲学？插件/钩子机制是如何支撑的？
- 资源管理（连接池、内存、线程模型）的策略和权衡？

### 库 / SDK / 工具

你是一个库设计和开发者体验专家。你对这个项目，想弄清楚：

- 核心抽象模型是什么？为什么选择这个抽象层次？
- 对外 API 的设计理念和一致性原则？
- 扩展性如何实现？用户可以在哪些点介入自定义行为？
- 错误处理和异常传播的策略？
- 性能优化的关键点？哪些设计是为了性能做了特殊处理？

### 移动应用 / 桌面应用

你是一个客户端架构专家。你对这个项目，想弄清楚：

- UI 状态管理和数据流的核心设计？为什么选择这种架构？
- 离线能力和数据同步策略？
- 导航和路由的设计，深层链接如何处理？
- 平台特定代码的组织方式？跨平台逻辑如何共享？
- 性能关键场景（列表滚动、图片加载、动画）的优化策略？

### 通用模板（不确定类型时使用）

你是一个资深软件工程师，擅长快速理解陌生代码库的设计意图。你对这个项目，想弄清楚：

- 项目解决的核心问题是什么？它的独特价值在哪里？
- 最核心的业务逻辑集中在哪里？这些逻辑为什么复杂？
- 模块间的依赖关系和通信方式？核心的数据流向？
- 项目中最重要的设计决策是什么？为什么这样决定？

## 约定发现规则

从 Phase 0 数据自动检测模式，生成约定发现方向。每条规则描述检测条件和对应的方向。

### 检测规则

| 检测条件 | 生成方向 |
|---------|---------|
| ≥5 个 Controller/Handler 类 | 探索 API 层的通用写法和约定：URL 命名、参数校验、返回值包装、异常统一处理 |
| ≥3 个 Service 实现类 | 探索服务层的约定：事务管理、接口vs实现分离、业务逻辑组织方式 |
| 发现异常类继承体系（≥2 层） | 探索异常处理规范：异常分类、错误码体系、全局异常处理器 |
| ≥3 个配置文件（yml/properties/conf） | 探索配置管理约定：多环境配置、敏感信息处理、配置优先级 |
| ≥5 个 DTO/VO/Request/Response 类 | 探索数据传输对象的使用约定：命名规则、字段校验、对象转换方式 |
| 发现拦截器/过滤器/中间件 ≥3 个 | 探索请求处理管道：拦截顺序、职责划分、通用处理（认证/日志/限流） |
| 发现枚举类 ≥5 个 | 探索枚举的使用模式：状态码、类型定义、枚举与数据的映射方式 |
| 发现定时任务/调度 ≥3 个 | 探索定时任务的约定：调度方式、并发控制、失败重试、幂等性 |
| 发现数据库迁移文件 | 探索数据库变更管理：迁移策略、版本管理、回滚机制 |
| ≥2 种消息/事件机制 | 探索异步通信模式：消息队列使用、事件发布/订阅、最终一致性 |

### 规则使用方式

1. 读取 `p0-data/structure.json` 和 `p0-data/hotspots.json`
2. 对每条规则检查数据中是否满足检测条件
3. 满足条件的规则生成对应方向，标记为 `type: "convention"`
4. 约定方向优先级低于领域专家方向，但同样进入探索队列

## 方向格式标准

每个方向必须包含以下字段：

```json
{
  "id": "d1",
  "type": "domain | convention",
  "question": "用问题的形式描述探索目标",
  "value": "回答这个问题能带来什么价值，为什么 AI 需要知道这个",
  "suggested_areas": ["建议探索的代码区域/目录"],
  "status": "pending",
  "priority": 1
}
```

### 好的方向 vs 差的方向

**好的方向**（面向"为什么"）：
- "订单状态机的完整流转规则是什么？哪些操作会触发状态变更？"
- "消息推送如何保证不丢不重？涉及哪些关键组件？"
- "数据权限的隔离粒度是如何实现的？行级过滤在哪个层面生效？"

**差的方向**（面向"是什么"，表层信息）：
- "项目有哪些模块？" → 看目录结构就知道
- "用了什么数据库？" → 看依赖就知道
- "API 有哪些接口？" → 看 Controller 就知道
